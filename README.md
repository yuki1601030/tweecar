# TweeCar プロトタイプ

TweeCar は、街中の車たちが裏側で匿名の状況情報を共有し、自車の AI が「今この運転手に必要」と判断した情報だけを運転中に届ける体験を検証するための、ユーザーテスト用の空想プロトタイプです。

## 目的

- 「車たちのつぶやき」が街全体で集まり、車専用 SNS のような情報基盤になる世界観を伝える。
- ドライバーが他車投稿を読むのではなく、自車 AI から必要な情報だけをタイミングよく受け取る利用シーンを体験してもらう。
- 明日のユーザーテストで、サービスの世界観と運転中の使い方を短時間で理解できる状態にする。

## 2画面構成

- **サービスイメージ**
  具体的な街の中で多くの車がつぶやき、投稿カードや吹き出しとして集まっていく様子を見せます。TweeCar AI がその中から重要情報を拾い、ドライバーへ届ける世界観説明用の画面です。

- **ユーザー体験**
  運転中の車載ナビを見ながら、自車 AI が自発的に重要情報を通知する体験画面です。右側のスマホには、音声通知や短い返答が履歴として連動します。

## 重要な前提

- ドライバーは他車の投稿を直接見ません。
- サービスイメージ画面の吹き出しは、世界観を伝えるための概念表現です。
- 実際の利用体験では、TweeCar AI が必要な情報だけを要約して通知します。
- 外部 API、ログイン、DB は使わず、すべてダミーデータで動作します。

## GitHub Pagesでの確認方法

`index.html` 1ファイルで完結する静的プロトタイプです。ビルドは不要です。

1. このリポジトリを GitHub に push します。
2. GitHub の **Settings → Pages** を開きます。
3. Source を対象ブランチ、Folder を `/ (root)` に設定します。
4. 発行された GitHub Pages URL を開くと、そのままプロトタイプを確認できます。

ローカルでは `index.html` をブラウザで直接開くか、任意の静的サーバーで確認できます。

## 明日のユーザーテスト用シナリオ版

このプロトタイプは、TweeCar のユーザーテスト用に、A車の運転者として高速道路を走っている感覚で操作できる車載ディスプレイ風 UI にしています。

### 体験設計の前提

- 車同士の生ツイートは人間には見せない設計です。
- ユーザーは A車の運転者として、音声通知と短いカードで状況を知ります。
- 画面端には「車同士の非公開SNSデータ通信」を抽象データとして表示し、裏側で情報共有が動いている世界観だけを伝えます。
- 「内部投稿を見る」トグルをオンにした場合のみ、開発者デモ表示として車たちの内部投稿を確認できます。
- PC 音声読み上げはブラウザの `speechSynthesis` を使い、`ja-JP` / `rate = 0.9` / `volume = 1` で再生します。

### 実装シーン

- **状況A：後方からの高速接近車両を事前回避する体験**
  B車が約140km/hで後方から接近し、A車AIが関連情報だけを要約して通知します。ユーザーは「注意を聞く」で音声通知を聞き、「走行車線に避ける」を押して危険を事前回避します。

- **状況B：後続車のながら運転を事前に察知して距離を取る体験**
  F車のながら運転リスクを車同士の非公開通信からA車AIが判断します。ユーザーは「注意を聞く」で音声通知を聞き、「追越車線へ移動」または「距離を取る」で安全距離を確保します。

### ユーザーテストで確認すること

- AI通知カードと音声通知が短く、運転中でもわかりやすいか。
- 生ツイートを見せずに、車同士が裏側で情報共有している世界観が伝わるか。
- 自車AIが必要情報だけを届けることに信頼感があるか。
- Privacy Guard の表現で、個人情報や生ツイートが守られている安心感があるか。
- 通知を聞いたあと、ユーザーが自然に「避ける」「距離を取る」など運転行動を変えられるか。
